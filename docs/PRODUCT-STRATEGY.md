## 一、当前所处的位置（诚实评估）

产品现状：一个 GitHub 高星仓库的排名/发现站（starrank.dev），8 语言、上线在 Vercel，后端在 droplet。

**核心问题：目前没有护城河。** 纯展示层面，OSS Insight / star-history / bestofjs 都做过了。差异化必须来自**别人没有的信号**，而这些大多还没建。

**优先级判断：不该碰变现，该做"差异化 + 一个能自传播的增长回路"。**
变现的正确顺序是 **差异化信号 → 流量 → 再变现**。低流量、无差异化阶段谈变现是本末倒置——AdSense 会被拒、赞助没受众、真正值钱的数据/API 还没做出来。

## 二、四个方向的判断

### 增长/流量（最该投入）
- **本品类唯一验证过的增长飞轮 = 可嵌入 README 的东西**（star-history 靠这个赢的）。每次嵌入 = 一个反向链接 + 一次免费曝光，永久挂着。
- **推荐做"排名徽章/卡片"而非克隆 star 曲线**：star 曲线 star-history 已占，但"在 AI 分类里排第几""本周上升 3 名"只有我们有（有归类+趋势）。心理钩子也更强（"我冲进前 20 了"）。
- 技术：用 SVG 纯数据生成（不用截图），`/badge/{owner}/{repo}.svg`，明暗两版，详情页放"复制到 README"按钮。
- **诚实前提**：徽章是"有了初始流量后的放大器"，不是从 0 到 1 的第一推动力——冷启动阶段没人知道你，也就没人来挂。0→1 仍要靠内容（HN 文章、社媒）。
- 其他渠道：SEO 长尾（复利慢热）、HN/Reddit/X（尖峰）、公众号/小红书（围墙花园，做站内内容可以，引流回站很弱）。

### 功能/差异化（决定有没有护城河）
按含金量排：
1. **刷 star 检测 / star 质量分** —— 最强支点。同时解决三件事：① 社媒选品的防翻车关卡 ② 传播爆点（"这个爆火项目 40% star 是刷的"能疯传）③ 唯一有人愿付钱的信号（VC 尽调、选型）。市面几乎没人认真做。
2. **归类导航**（已 66%）—— vs 纯 trending 站的现成差异点，值得打磨。
3. **开发者图谱**（repo→背后的人）—— 中长期，见第三节。

### 变现（现在是干扰项）
- AdSense：需流量+原创内容+解决国内访问，现在申请大概率被拒。
- **真正的钱在数据/API**：刷 star 检测、依赖风险评分这类独家信号卖给 VC/企业，付费意愿最高。但要求先把信号做出来。

### 优化/技术债（别拖成雷）
- **32 个依赖漏洞**（7 critical）—— 主要旧 NestJS package.json，清一轮。
- **MongoDB 无自动备份** —— 单节点 droplet，数据丢了全没，风险最实在。
- **ClickHouse 回填报错** —— 日志一直刷，待查。
- **Vercel 国内访问** —— 一直没解，重视国内流量迟早要面对。

## 三、重点方向：X 用户 × GitHub 开发者（讨论最深的一块）

### 想法演进
从"给 repo 富集 X 数据" → 演进到 **"把值得关注的开发者做成产品：按领域分类、排名，帮人发现该关注谁"**。

### 价值判断：值得做，但有两条铁律
1. **必须按实绩排，不能按粉丝排。** 按粉丝 = 满大街的大路货目录，废物。按"他做过什么"（GitHub 实绩）排 = 只有我们能做，因为没人把 GitHub 和 X 两张图连起来。**"某领域最能造东西的人 + 他的 X"** 这个排序逻辑是护城河。
2. **老实定位成"开发者/技术圈"，别吹成全行业。** 我们只认识"会写代码的人"，对设计师/VC/财经 KOL 零信号。**"窄"在这个阶段是特性不是缺点**——窄+深+有护城河，碾压宽+浅+大路货。

### 关于"一个行业一个行业做"
- **别按粉丝铺开** —— 每加一个没有实绩信号的行业，就把自己稀释成大路货一次。
- **正确扩法：每个行业自带实绩信号源**（AI 研究者→arXiv/Scholar，Maker→Product Hunt，设计师→Dribbble，Crypto→链上+GitHub），保持"用真实作品排名"的统一内核。但每个新行业 = 一整套新数据接入，是重活。
- **先深后广、沿信号扩**：先把"开发者/AI 圈该关注谁"做到权威（AI 是当下注意力最集中、付费意愿最高的赛道，不算窄），再扩相邻可复用信号的垂直，别跳到"离得很远"的行业。
- 判断标准不是"覆盖多少行业"，而是"你是不是某一个问题的权威答案"。

### 留存坑
"关注清单"是一次性消费。解法：做成**动态榜**（"本月新晋值得关注的 AI builder""这周谁影响力在涨"），而非静态 top 列表。

### 第三方 X API 分析（RapidAPI / twitterapi.io / SocialData / Apify / Bright Data 等）
- **它们把 X 从"碰不起"变成"targeted 场景用得起"**：官方 Pro 档 $5000/月 → 这些按量约 $0.1–0.5/千条，定向用途几十刀/月。（价格常变，下单前现查。）
- **代价从钱换成 ToS/稳定性**：它们都是违反 X 条款在爬。provider 会被 X 封杀导致管道不可预测地断；**打包转售爬来的 X 数据法律暴露最大**。
- **聪明架构**：让 GitHub 提供【身份关联】（`twitter_username`，用户自声明、免费、合规），第三方 API 只查【已知 handle 的公开指标】（粉丝数、近期动态）。绕开最贵最脏的"身份识别+图谱爬取"。
- **用/不用**：内容素材、富集粉丝数、200 个人工种子 → 可用（断了不影响）；做成常在的核心支柱、转售 → 别。

## 五、落地第一步（想做时从这里开始）

**建"开发者层"**——这是 X/影响力/关注榜等一切的地基：

1. 新建 `developers` 集合，遍历 owner → 调 GitHub `/users/{login}` → 存 `type / followers / twitter_username / bio / company / htmlUrl`。
2. 44,988 个 owner，GitHub 认证额度 5000/小时，约 9 小时跑完；或先只跑高星 repo 的 owner，几十分钟出个能用的种子。
3. 排名主体**必须过滤掉 Organization**（`type` 字段区分），否则前排全是 facebook/google。
4. 有了开发者层，再叠：自声明 twitter 富集 → 第三方 API 补 X 侧动态 → 按技术领域分类的"值得关注的 builder + 他们的 X" → 动态榜。
