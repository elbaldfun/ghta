# 内容选题清单

引流用的原创分析选题。挑选原则：**只写别人写不出来的**——护城河是 6.7 万仓库的语料和逐日快照，不是观点。

发布方式见 [`web/content/blog/`](../web/content/blog/)：Markdown 文件，支持 ` ```starrank:xxx ` 围栏块嵌入实时数据。

> 编号（`T01`…）是**稳定 ID**，只用来指代，不表示优先级——优先级会随流量反馈调整，见文末「当前推荐顺序」。

## 三个维度

每个选题用三个独立维度描述，不要混为一谈：

- **就绪度** — `可写` / `等数据` / `需开发`
- **引流类型** — `爆发型`（HN/Reddit 一次性大流量）/ `长尾型`（SEO 复利，慢热）
- **难度** — 低 / 中 / 高（指实现成本，不是写作难度）

## 选题详情

### T01 · 我们如何追踪 6.7 万个 GitHub 仓库
抓取分片、GraphQL 配额管理、时序集合选型，以及踩过的坑（时序集合的 `meta` 索引问题让指标任务从 10 分钟变成 129 分钟）。
**为什么有效**：HN 长期偏爱基础设施/方法论文章。**同时是 AdSense 审核最认可的"关于本站"类原创内容**，一篇顶两个用途。不需要任何新数据。

### T02 · 19% 的高星项目你在法律上不能用
12,706 个高星仓库**没有任何许可证**——默认保留所有权利，企业不能合法使用。
数据：MIT 24,484 / 无 12,706 / Apache 10,639 / GPL-3.0 4,487 / BSD-3 1,567。
**为什么有效**：结论有实际后果，且多数开发者没意识到。延伸角度：按语言拆，哪个社区最不在意许可证。
**需要**：新增 `/stats/licenses` 端点。

### T03 · 高星榜上有多少根本不是软件？
榜首大量是 awesome-list、教程、面试题库，而非可运行项目。
**为什么有效**：直接质疑 star 榜这个行业通用指标的有效性。
**注意**：只能用 topic + 无许可证 + 无主语言做**启发式**判定，必须在文中如实声明，否则容易被挑错。

### T04 · 那些正在衰落的明星项目
用 `star_history` 的十年曲线，找增速持续下滑甚至停滞的高星项目。
**为什么有效**：**所有人都写"增长最快"，没人写"正在死去"**，角度稀缺。
**注意**：措辞要克制——项目成熟后增速放缓是正常现象，不等于衰落。判定标准必须写清楚。
**需要**：按 `star_history` 算长期增速的端点。

### T06 · Fork/Star 比揭示的项目类型
高 fork 低 star = 模板/教学；高 star 低 fork = 工具/参考。给出"这项目是拿来用的还是拿来学的"判断法。

### T07 · Issue 积压排行
`openIssues / stars` 比值，对技术选型有实际参考价值。
**注意**：issue 多不等于维护差，有的项目把 issue 当讨论区，需说明。

### T08 · GitHub 上的中文开源项目图景
按 README/描述含中文筛选，看规模、领域分布、头部项目。
**为什么值得做**：中文搜索场景**竞争极低**，且天然适合国内社区传播。

### T09 · 本周爆发项目追踪
需 `weeklyIncrease`。**建议做成固定周更栏目**——传播力最强，且能培养回访。

### T15 · 分类生态盘点
**当前写不了**：只有 1.6% 的仓库有分类。前置是解决归类——把 `AI_PROVIDER` 换成云端 LLM（现在指向局域网的 LM Studio，droplet 访问不到）。

## 当前推荐顺序

1. **T01 方法论** —— 投入最低，建立可信度，直接利好 AdSense 审核
2. **T02 许可证** —— 发现最扎实，标题自带冲突感
3. **T04 衰落项目** —— 角度稀缺，需要一点开发
4. **T09 周爆发榜** —— 等 7/25 数据齐，转固定周更

前两篇发出去后看真实流量反馈，再决定加码哪个方向。**不建议一次写太多**——先验证哪类内容真的能带量。

- [ ] 中英两版都写（`content/blog/zh/` 与 `content/blog/en/`）
- [ ] 文末如实写明数据边界——样本是"已流行项目"而非全量，star ≠ 质量
- [ ] 能用 ` ```starrank: ` 实时数据块的地方就别贴死数字
