## 第一梯队：现在就能写，且有传播力

这几篇的共同点是**有一个反直觉的、能一句话说清的发现**——这是能上 HN / Reddit 的前提。

### A1. 19% 的 GitHub 高星项目，你在法律上不能用

- **发现**：12,706 个高星仓库**没有任何许可证**。无许可证 = 默认保留所有权利 = 企业不能合法使用。
- **为什么能传播**：结论有实际后果，而且大多数开发者没意识到。标题自带冲突感。
- **数据**：✅ 完全就绪（MIT 24,484 / 无 12,706 / Apache 10,639 / GPL-3.0 4,487 …）
- **延伸**：按语言拆——哪门语言的社区最不在意许可证？
- **渠道**：HN、Reddit r/programming、法务/合规向的 newsletter
- **难度**：低。需要新增一个 `/stats/licenses` 端点。

### A2. 高星榜上有多少根本不是软件？

- **发现**：star 榜前列大量是 awesome-list、教程、面试题库，而非可运行的项目。
- **为什么能传播**：直接质疑"star 榜"这个行业通用指标的有效性，观点鲜明。
- **数据**：✅ 就绪。可用 topic（`awesome`/`list`/`tutorial`/`books`）+ 无许可证 + 无主语言等特征做启发式判定，并**如实说明这是启发式而非精确分类**。
- **渠道**：HN（这类"指标批判"很吃香）
- **难度**：中。判定规则要谨慎，避免误伤。

### A3. 我们如何追踪 6.7 万个 GitHub 仓库

- **内容**：抓取分片、GraphQL 配额管理、时序集合选型、以及踩过的坑（时序集合的 `meta` 索引问题让指标任务从 10 分钟变成 129 分钟）。
- **为什么能传播**：HN 长期偏爱基础设施/方法论文章。**同时是 AdSense 审核最喜欢的"关于本站"类原创内容**，能显著提升可信度。
- **数据**：✅ 不需要数据，写的是系统本身。
- **渠道**：HN、lobste.rs
- **难度**：低。**建议优先写这篇**——投入产出比最高，且对广告审核直接有帮助。

### A4. 那些正在衰落的明星项目

- **发现**：用 `star_history` 的十年曲线，找出增速持续下滑、甚至停滞的高星项目。
- **为什么能传播**：**所有人都在写"增长最快"，没人写"正在死去"**。稀缺角度，讨论度高。
- **数据**：✅ 3,018 个仓库有长期曲线，够写。
- **注意**：措辞要克制——项目成熟后增速放缓是正常的，不等于"衰落"。把标准写清楚，否则容易招致反弹。
- **渠道**：HN、X
- **难度**：中。需要新增按 `star_history` 算长期增速的端点。

## 第二梯队：SEO 长尾，慢热但复利

### B1. 语言生态对比 ✅ 已发布

### B2. Fork/Star 比揭示的项目类型

- 高 fork 低 star = 模板/教学；高 star 低 fork = 工具/参考。可据此给出"这个项目是拿来用的还是拿来学的"判断法。
- **数据**：✅ 就绪。**难度**：低。

### B3. Issue 积压排行：哪些热门项目维护压力最大

- `openIssues / stars` 比值。对技术选型有实际参考价值（选型时要看维护状况）。
- **数据**：✅ 就绪。**难度**：低。
- **注意**：issue 多不等于维护差，有的项目把 issue 当讨论区。需说明。

### B4. GitHub 上的中文开源项目图景

- 按 README/描述含中文筛选，看规模、领域分布、头部项目。
- **为什么值得做**：中文搜索场景**竞争极低**，且天然适合在国内社区传播。
- **数据**：✅ 就绪（README 已入库）。**难度**：中。

## 建议顺序

1. **A3 方法论** —— 投入最低，建立可信度，直接利好 AdSense 审核
2. **A1 许可证** —— 发现最扎实，传播力强
3. **A4 衰落项目** —— 角度稀缺，需要一点开发
4. **C1 周爆发榜** —— 等 7/25 数据齐了，转为固定周更

前两篇发出去后看真实流量反馈，再决定加码哪个方向。**不建议一次写太多**——先验证哪类内容真的能带量。
