# 内容选题清单

引流用的原创分析选题。挑选原则：**只写别人写不出来的**——护城河是 6.7 万仓库的语料、逐日快照，以及 4,241 个仓库回溯到 2015 年的长期曲线。

发布方式见 [`web/content/blog/`](../web/content/blog/)：Markdown 文件，支持 ` ```starrank:xxx ` 围栏块嵌入实时数据。

> 编号（`T01`…）是**稳定 ID**，不表示优先级。
> 三个独立维度：**就绪度**（可写 / 等数据 / 需开发）、**引流类型**（爆发型 / 长尾型）、**难度**。

## 爆款候选（按传播潜力排序）

### T20 · 你收藏的高星项目，40% 已经一年没人动了

- **发现**：27,006 个高星仓库超过一年没有 push，20,435 个超过两年。
- **爆款理由**：数字大、反直觉、**对技术选型有直接后果**。标题自带"你以为的和实际的"张力。这是目前最强的一个。
- **就绪度**：可写 · **引流类型**：爆发型 · **难度**：低
- **需要**：`/stats/staleness` 端点（按 `pushedAt` 分桶）
- **注意**：稳定的成熟库长期不更新是正常的（比如已经"写完了"的工具）。必须区分「废弃」和「稳定」，否则会被专业读者打脸。建议同时给出 issue 活跃度做交叉验证。

### T21 · 19% 的 GitHub 高星项目，你在法律上不能用

- **发现**：12,706 个仓库没有任何许可证——默认保留所有权利，企业不能合法使用。
- **爆款理由**：有法律后果，而多数开发者没意识到。合规/法务圈也会转。
- **就绪度**：可写 · **引流类型**：爆发型 · **难度**：低
- **需要**：`/stats/licenses` 端点
- **延伸**：按语言拆——哪个社区最不在意许可证。

### T22 · Star 通胀：2015 年的 1000 star，相当于今天的多少？

- **发现**：用 4,241 个仓库的十年曲线，算出不同年份获得同等 star 数的难度差异。
- **爆款理由**：**这个角度几乎没人做过**，且能给出一个可引用的"换算系数"。数据独家性最强。
- **就绪度**：可写 · **引流类型**：爆发型 · **难度**：中
- **需要**：基于 `star_history` 的时间序列分析端点
- **注意**：GitHub 用户总量本身在增长，需要在文中说明这是相对而非绝对结论。

### T23 · 我们如何追踪 6.7 万个 GitHub 仓库

- **内容**：抓取分片、GraphQL 配额管理、时序集合选型，以及踩过的坑（时序集合 `meta` 索引问题让指标任务从 10 分钟变 129 分钟）。
- **爆款理由**：HN 长期偏爱基础设施/方法论文章。**同时是 AdSense 审核最认可的"关于本站"类原创内容**，一篇顶两个用途。
- **就绪度**：可写（**不需要任何新端点**）· **引流类型**：爆发型 · **难度**：低

### T24 · 高星榜上有多少根本不是软件？

- **发现**：榜首大量是 awesome-list、教程、面试题库，而非可运行项目。
- **爆款理由**：直接质疑 star 榜这个行业通用指标的有效性，观点鲜明。
- **就绪度**：可写 · **引流类型**：爆发型 · **难度**：中
- **注意**：只能用 topic + 无许可证 + 无主语言做**启发式**判定，必须在文中如实声明。

### T25 · 那些正在死去的明星项目（点名）

- **发现**：用长期曲线找增速持续下滑甚至停滞的高星项目。
- **爆款理由**：**所有人都写"增长最快"，没人写"正在死去"**。点名会带来争议度，争议带流量。
- **就绪度**：可写 · **引流类型**：爆发型 · **难度**：中
- **风险**：**这篇最容易招反弹**。措辞必须克制，判定标准写死在文里，最好给被点名项目留出"可能的解释"。

### T26 · README 写多长，才能拿到更多 star？

- **发现**：60,453 份 README 全文在库，可分析长度、结构（有无徽章/示例/目录）与 star 的相关性。
- **爆款理由**：**可操作**——读者看完能立刻改自己的 README。这类"你能拿去用"的内容分享率高。
- **就绪度**：可写 · **引流类型**：爆发型 + 长尾型 · **难度**：中
- **注意**：相关不等于因果，必须明说。

### T27 · 选 GPL 的项目，是不是更容易被弃坑？

- **发现**：交叉 `license` 与 `pushedAt`，看不同许可证的项目存活率差异。
- **爆款理由**：许可证之争是开发者永恒话题，用数据切入很新鲜。
- **就绪度**：可写 · **引流类型**：爆发型 · **难度**：中
- **风险**：容易被解读成立场之争，结论要写得非常克制。

## 当前推荐顺序

1. **T23 方法论** —— 零开发成本，建立可信度，直接利好 AdSense 审核
2. **T20 40% 没人维护** —— 数字最有冲击力，选型场景强相关
3. **T21 19% 不能合法用** —— 法律后果，受众外扩到合规圈

前两篇发出去后看真实流量，再决定加码方向。**不建议一次写太多**——先验证哪类内容真的能带量。

## 发布前检查

- [ ] 文件名只能用**小写字母、数字、连字符**（slug 白名单 `^[a-z0-9-]+$`，否则 404）
- [ ] frontmatter 的 `title`/`description` 若含**半角冒号，必须加引号**（否则 YAML 解析失败，文章静默消失）
- [ ] 译文加 `translatedFrom: en`（会自动显示机翻提示 + 原文链接）
- [ ] 文末如实写明数据边界——样本是"已流行项目"而非全量，star ≠ 质量
- [ ] 能用 ` ```starrank: ` 实时数据块的地方就别贴死数字

## 查数据时的坑

- **`pushedAt` 是 BSON Date，不是字符串**。用字符串比较会**静默返回 0**（我第一次就踩了），必须用 `new Date()` 比较。
- `metric_snapshots` 是 timeseries 集合，查询要走 `meta.externalId` 才能命中索引。
